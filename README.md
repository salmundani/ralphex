Ralphex is a Claude Code plugin that creates an automated development loop between two AI systems: **Claude Code** (planner + implementer) and **OpenAI Codex** (code reviewer). The loop repeats until Codex approves the code with LGTM.

I usually prefer running a single agent model, but from my coding sessions, I feel that Claude is really good at planning and executing, while Codex really excels at code review.

## Installation

### Prerequisites

- [Claude Code](https://code.claude.com/docs)
- [OpenAI Codex CLI](https://github.com/openai/codex) installed and authenticated

### Install from marketplace

```shell
/plugin marketplace add salmundani/ralphex
/plugin install ralphex@salmundani-ralphex
```

### Usage

Once installed, run the skill from Claude Code:

```shell
/ralphex:ralphex <your task description>
```

## How it works

I was constantly doing the following loop manually, this plugin just automates it:

```
Prompt вҶ’ Plan вҶ’ Implementation вҶ’ Commit вҶ’ Codex Review вҶ’ Claude filters reviews вҶ’ End
         вҶ‘  вҶ“         вҶ‘                                             вҶ“
         в””в”Җв”Җв”ҳ         в””в”Җв”Җв”Җв”Җв”Җв”Җ loop if there are corrections в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

## Step 0: Read Settings

Before any work begins, Ralphex loads (or creates) a per-project settings file at `.claude/ralphex.local.md`. This file stores three values as YAML frontmatter:

- **`base_branch`** вҖ” The branch to diff against (e.g. `main`).
- **`codex_model`** вҖ” The OpenAI model Codex will use for reviews (e.g. `gpt-5.2-codex`).
- **`codex_reasoning_effort`** вҖ” How hard Codex thinks during review (`low`, `medium`, `high`, `xhigh`).

If the file doesn't exist, the user is prompted to provide these values and the file is created.

Finally, Ralphex resolves the **review target** вҖ” the git ref that Codex will diff against. If you're on a feature branch, this is `base_branch`. If you're already on the base branch, it falls back to `origin/{base_branch}`. If there's no diff at all, the workflow stops early.

## Step 1: Plan

### First iteration вҖ” Structured planning with user approval

The first time through the loop, Claude Code goes through four phases:

1. **Initial Understanding** вҖ” Explores the codebase (Glob, Grep, Read) to understand the project structure, existing patterns, and where the task fits in.
2. **Design** вҖ” Identifies which files to create or modify, what patterns to follow, and what existing code to reuse.
3. **Review** вҖ” Self-reviews the design for soundness, checks alignment with the user's request, and asks clarifying questions if needed.
4. **Final Plan** вҖ” Presents the plan to the user and waits for explicit approval before proceeding. The user can request changes.

### Subsequent iterations вҖ” Lightweight re-plan

After the first cycle, re-planning is fast and automatic (no user approval needed). Claude Code only addresses the specific corrections accepted from the Codex review, doing focused exploration only when those corrections reference code not seen before.

## Step 2: Implement

Claude Code executes the plan вҖ” writing, editing, and creating files as needed. This is standard Claude Code implementation work, with full access to the codebase tools (Read, Write, Edit, Grep, Glob, Bash).

## Step 3: Commit

A clean commit is created so Codex has something to review:

1. Stage specific changed files.
2. Verify the working tree is clean with `git status`.

The workspace **must** be completely clean before proceeding вҖ” Codex reviews the committed diff, so uncommitted changes would be invisible to it.

## Step 4: Codex Code Review

Ralphex invokes Codex CLI to review the branch diff:

```bash
codex exec \
  --sandbox read-only \
  -m {codex_model} \
  -c model_reasoning_effort="{codex_reasoning_effort}" \
  -o .claude/ralphex-review.txt \
  "Review the changes of this branch against {review_target}. If you find issues, bugs, improvements, or corrections, describe each one clearly. If the code looks good and you have no corrections, respond ONLY with the exact text: LGTM"
```

Key details:
- Codex runs in **read-only sandbox** mode вҖ” it can read the repo but can't modify it.
- The review output is written to `.claude/ralphex-review.txt`.
- If the command fails (non-zero exit), the workflow stops and the user is informed.

## Step 5: Evaluate Review

Claude Code reads the review file and evaluates the result:

### LGTM (clean review)

If Codex responds with just "LGTM", the implementation is approved. Ralphex reports success, shows the total iteration count, cleans up the review file, and stops.

### Corrections found

If Codex lists issues, Claude Code **critically evaluates each suggestion** independently rather than blindly accepting everything. Each suggestion is classified as:

- **Accept** вҖ” Real bugs, security issues, correctness problems. These get addressed in the next iteration.
- **Reject** вҖ” Factually wrong suggestions, misunderstandings, or purely stylistic nitpicks.
- **Defer** вҖ” Valid observations that are out of scope for the current task (e.g. pre-existing issues).

The verdicts are displayed to the user:

```
Iteration 1 - Codex Review Evaluation:
- вң… Accept: Missing null check in handleResponse()
- вқҢ Reject: Codex misread the variable scope
- вҸӯпёҸ Defer: Unrelated function could use better error handling
```

Then:
- If **all suggestions are rejected or deferred** вҶ’ treated as a clean review. Done.
- If **any suggestions are accepted** вҶ’ the review file is deleted and the loop goes back to **Step 1** (lightweight re-plan) with only the accepted corrections as context.

The loop continues automatically until Codex gives LGTM or all remaining suggestions are rejected/deferred.
